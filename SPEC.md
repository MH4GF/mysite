## テスト戦略

### 基本原則

- テストカバレッジ 100% は must 要件である。ただし単一の数値ではなく、対象領域ごとに直交する 3 つのゲートに分解して計測する。すべてのゲートは CI で強制され、ローカルでも同一コマンドで検証できる
- カバレッジの例外は中央の allowlist ファイルでのみ管理する。インラインの ignore コメント（`/* v8 ignore */` 等）は全面禁止とする。例外の追加は必ず diff に現れ、理由コメントを必須とし、人間の承認を要する
- テストは「実行されたか」ではなく「壊れたら検知されるか」を担保する。行カバレッジのゲーミング（アサーションのない空テスト）はミューテーションテストで構造的に無効化する

### 3 つのゲート

| ゲート | 対象 | 実行体 | 閾値 |
|---|---|---|---|
| ① 行カバレッジ | ロジック + 全コンポーネント | Vitest（unit + portable stories in Browser Mode） | lines / branches / functions / statements すべて 100% |
| ② ルートカバレッジ | 全ルート | Playwright ページVRT + axe | 全ルートが VRT 対象に登録されていること（100%） |
| ③ ミューテーションスコア | PR の変更ファイル | StrykerJS（incremental モード） | 80% 以上 |

### テストレイヤの割り当て

各コードは以下のいずれかのレイヤで必ず検証される:

- **純ロジック**（ユーティリティ、データ変換、markdown パイプライン等）: Vitest ユニットテスト。`**/__tests__/**/*.test.ts?(x)` に配置する
- **クライアント / 共有コンポーネント**: Storybook のストーリーを唯一の情報源とし、portable stories として Vitest Browser Mode（Playwright プロバイダ、実ブラウザ）で実行する。ストーリーはゲート①の行カバレッジに算入されるため、ストーリーのないコンポーネントは CI で検出される
- **RSC・ページ・レイアウト**: ゲート②のページVRTで検証する。ゲート①からは allowlist で除外する（除外理由: ゲート②で 100% 計測されるため）
- **横断フロー**（複数ページ・エンドポイントをまたぐ振る舞い）: Playwright E2E テスト（`app/**/*.e2e.ts`）

### コンポーネントテスト（Storybook）

- すべてのクライアント / 共有コンポーネントは対応する stories ファイルを持つ。存在は CI の構造チェックで検証する（コンポーネントファイルと stories ファイルの突合。除外は理由コメント付き allowlist で管理する）
- 1 つのストーリーは 4 役を兼ねる: コンポーネントテスト（play function によるアサーション）、コンポーネントVRT、a11y 検査（axe）、行カバレッジ
- コンポーネントの状態（開閉、ローディング、エラー、フォールバック等）は play function を用いて「状態 = ストーリー」として列挙する。デフォルト表示のみのストーリーで済ませてよいのは状態を持たないコンポーネントに限る
- 外部依存（HTTP、外部サービス）は MSW で必ずモックし、ストーリーは完全に決定的であること。実ネットワークに依存するストーリーを書いてはならない
- コンポーネントVRT は全ストーリーに対して自動実行される（ランナーのフックで無条件に撮影する）。ストーリーごとの opt-in 方式にしてはならない。これにより「ストーリーが存在する ⇒ VRT が存在する」が定義上成立する
- スクリーンショットはストーリー単位で管理し、VRT 差分はコンポーネント単位でレビューする
- a11y 検査は全ストーリーに対して axe を実行し、violation ゼロを must とする

### 視覚的状態行列（VRT の検知範囲の定義）

- 視覚的状態空間は本来無限であるため、「視覚的デグレの検知 100%」は本セクションで宣言する状態行列に対して定義する。行列の全セルが自動撮影されることを機械的に保証する
- 軸の分担:
  - **コンポーネントVRT**: テーマ（ライト / ダーク）× 状態（props バリアント、インタラクション状態、疑似クラス）
  - **ページVRT**: ビューポート（Desktop / Mobile）× テーマ。レスポンシブ崩れ・z-index・overflow・カスケード等、コンポーネントの合成で初めて現れる問題を担う
- テーマ・ビューポート軸はランナー側の設定で全ストーリー / 全ページに直交適用する。個別のストーリー・テストコードに軸の適用を書かせてはならない
- hover / focus 等の疑似クラス状態は pseudo-states アドオンで状態ストーリーとして列挙する
- CSS 起因の状態（疑似クラス、ダークモードの CSS 分岐、ブレークポイント）はコードカバレッジに写らない。したがってこれらの網羅はカバレッジではなく、上記の軸の機械適用によって保証する
- ストーリー作成者（人間・エージェント）に残る列挙責務は「props とインタラクション状態」のみとする

### VRT の決定性

- 偽陽性率は検知率を直接毀損する（環境ノイズの差分はレビューのラバースタンプ化を招き、本物のデグレを素通りさせる）。したがって VRT の偽陽性の根絶は検知可能性の一部であり、must 要件である
- スナップショットは CI の単一環境（固定された Linux イメージ）でのみ生成する。ローカルで生成したスナップショットをコミットしてはならない
- 撮影時はアニメーションを全停止し、時刻・乱数・外部データはモックで固定し、フォントのロード完了を待つ
- スナップショット更新を含む PR には差分画像レポートを必ず添付し、差分は画像としてレビューする

### テスト実行モデル（二相実行）

- 「検知の 100% 保証」はリリース境界（main へのマージ〜デプロイ）に紐づけ、「フィードバック速度」は PR に紐づける。両者を同じタイミングで満たそうとしてはならない
- **PR / ローカル**: 変更の影響範囲のみ実行する（モジュールグラフに基づく affected 選別。`vitest --changed` 相当）。エージェントの自己検証ループはこの affected 実行で行う
- **全量実行トリガー**: モジュールグラフに写らない全域変更（Tailwind 設定、グローバル CSS、デザイントークン、フォント）はパスベースで明示的に指定し、該当時は PR でも全量実行する。このトリガーの漏れは affected 実行の検知の穴となるため、対象パスは設定として宣言的に管理する
- **リリース境界**: main へのマージ時に全行列を実行し、その成功をデプロイの前提条件とする
- 速度バジェット: PR のテストフィードバックは 5 分以内、全量実行は 15 分以内とする。超過した場合はシャーディング等の並列化・行列の軸分担の見直しによる最適化を先に行い、検知範囲（行列のセル）の削減は最後の手段とする。エージェントは実行時間を理由にテストの削減を提案してはならない

### ページVRT（Playwright）

- 役割は「RSC・レイアウト・contentlayer の結合の煙テスト + ページ全体の axe 検査」に限定する。コンポーネントの状態検証を持たせてはならない（それはストーリーの責務）
- 全ルートを対象とする。ライト / ダーク両モード × Desktop Chrome / Mobile Chrome で、フルページスクリーンショット比較と axe 検査を行う
- モック困難な外部依存要素（外部サービス由来の画像等）はマスクしてよい。マスクには理由コメントを付ける
- VRT 対象リスト（TargetPage）は独立したモジュールとして定義し、ゲート②の検証スクリプトから import 可能であること

### ルートカバレッジ検証（ゲート②）

- 検証スクリプトは `app/**/page.tsx` から全ルートを機械的に列挙する。動的ルート（`[slug]` 等）は contentlayer から代表 slug を解決する
- 列挙した全ルートが ページVRT の対象リストに含まれない場合、CI を fail させる
- ルートの除外が必要な場合は、ゲート①と同様に理由コメント付きの allowlist で管理する

### カバレッジ例外の管理（allowlist）

- ゲート①の除外は単一のファイル（`coverageExclusions.ts`）で管理する。エントリごとに除外理由をコメントで必須とする
- 正当な除外理由は以下のカテゴリに限る:
  - Next.js 規約ファイル（`page.tsx`, `layout.tsx`, `opengraph-image.tsx`, `sitemap.ts` 等）— ゲート②で担保されるため
  - 生成物・設定ファイル
  - ユニットテスト不能な外部サービス境界 — ページVRT / E2E で担保されるものに限る
- allowlist は縮む方向にのみ変更してよい（ラチェット）。新規追加はユーザーの明示的な承認を必要とする。AI エージェントは allowlist への追加を提案できるが、承認なしに適用してはならない

### ミューテーションテスト（ゲート③）

- StrykerJS を incremental モードで、PR の変更ファイルに限定して実行する
- ミューテーションスコアの閾値は 80% とする
- ミューテーションテストの除外・閾値変更にもユーザー承認を要する

### アーキテクチャ制約（テスト容易性のため）

- RSC はデータ取得と props の組み立てに徹し、表示ロジックはクライアント / 共有コンポーネントに委譲する。表示ロジックを RSC に置いてはならない（ストーリー化できず、ゲート①の対象から漏れるため）
- コンポーネントは外部依存を直接呼ばず、モック可能な境界（fetch 層、props）を経由する

### AI エージェントの作業契約

AI エージェント（サブエージェントを含む）がこのリポジトリで作業する際の契約:

1. コードを追加・変更したら、対応するレイヤのテスト（ユニット / ストーリー / ページVRT 登録）を同じ変更内で追加・更新する
2. 提出前に `pnpm test` を実行し、全ゲートがローカルで green であることを確認する。未カバー行・未登録ルートが出力された場合は自力で修正を繰り返す
3. allowlist（カバレッジ除外・ルート除外・ミューテーション除外）への追加は提案のみ可。ユーザー承認なしに適用しない
4. VRT 差分が出た場合、差分が自分の変更の意図した結果であることを説明できる状態でのみスナップショットを更新する。説明できない差分は報告して停止する
